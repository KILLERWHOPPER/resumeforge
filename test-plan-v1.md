## 5. 缺陷管理流程

### 5.1 严重度分级

| 级别 | 定义 | 响应时限 | 修复时限 |
|------|------|----------|----------|
| **Critical (P0)** | 核心流程阻断、数据丢失、安全漏洞、全站不可用 | 1 小时 | 4 小时 / 发布阻断 |
| **High (P1)** | 主要功能异常、性能严重劣化、兼容性阻断 | 4 小时 | 1 天 / 发布前必须修 |
| **Medium (P2)** | 次要功能异常、UI 不一致、非核心路径报错 | 1 天 | 3 天 / 下个 Sprint |
| **Low (P3)** | 体验优化、文档错误、代码规范、技术债 | 1 周 | 下个版本 / 积累重构 |

### 5.2 缺陷生命周期

```
NEW → ASSIGNED → IN_PROGRESS → FIXED → VERIFIED → CLOSED
  ↓                            ↑
  └──────── REOPENED ─────────┘
```

- **NEW → ASSIGNED**：QA 分配给对应开发
- **FIXED → VERIFIED**：开发自测 + QA 复测通过
- **REOPENED**：验证失败或引入新问题

### 5.3 阻塞发布标准

| 条件 | 处理 |
|------|------|
| 存在未修复 P0 | ❌ 绝对阻塞 |
| P1 未修复 > 2 个 | ❌ 阻塞 (PM 可豁免 1 个) |
| 测试覆盖率 < 80% (核心模块) | ❌ 阻塞 |
| 关键安全漏洞未修复 | ❌ 绝对阻塞 |
| Staging 冒烟测试失败 | ❌ 阻塞 |

## 6. 自动化测试基础设施

### 6.1 CI 集成点

| 阶段 | 执行内容 | 超时 | 失败策略 |
|------|----------|------|----------|
| **Pre-commit** | Ruff、ESLint、Prettier、MyPy、单元测试 (快) | 2 min | 阻塞提交 |
| **PR Checks** | 全量单测、集成测试、类型检查、构建 | 15 min | 阻塞合并 |
| **Nightly** | 全量 E2E、性能基线、安全扫描、视觉回归 | 60 min | 创建 Issue |
| **Release** | 冒烟测试、回归套件、兼容性矩阵、生产部署验证 | 30 min | 阻塞发布 |

### 6.2 测试报告与可视化

- **单测覆盖率**：Codecov / Coverage.py HTML 报告
- **E2E 报告**：Playwright HTML Report + Trace Viewer
- **性能基线**：k6 + Grafana 仪表板，自动对比 ±10% 阈值
- **安全扫描**：SARIF 上传 GitHub Security 标签页
- **视觉回归**：Percy/Chromatic 差异审批流
