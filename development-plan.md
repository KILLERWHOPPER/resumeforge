## 1. 项目全景

### 1.1 团队角色

| 角色 | Agent ID | 职责 |
|------|----------|------|
| 👤 产品经理 | `product-manager` | 需求管理、PRD、开发计划、验收 |
| 🎨 UI/UX 设计师 | `ui-designer` | 设计系统、界面原型、交互规范、PDF 模板设计 |
| ⚛️ Web 前端 | `web-frontend` | Next.js 前端开发、组件实现、编辑器集成 |
| 🐍 后端开发 | `backend-dev` | FastAPI 后端开发、数据库、LLM 集成 |
| 🧪 测试工程师 | `qa-engineer` | 测试用例、自动化测试、ATS 验证、性能测试 |
| 🔧 DevOps | `devops-engineer` | Docker Compose、CI/CD、部署、监控 |
| 🧠 协调 | `Friday / 皇堡` | 技术评审（CTO）、跨团队协调 |

### 1.2 迭代节奏

| 项目 | 取值 |
|------|------|
| Sprint 时长 | **2 周**（10 个工作日） |
| 总 Sprint 数 | **6 个**（Sprint 0 ~ Sprint 5） |
| 总工期 | **12 周** |
| 每日站会 | 15 分钟，同步进度与阻塞 |
| Sprint Review | 每个 Sprint 结束前周五下午 |
| Sprint Retro | Review 后 30 分钟 |
| 代码评审 | 每个 PR 必须有至少 1 人 Review |

### 1.3 交付原则

- **功能分支工作流**：每个功能/用户故事一个分支，命名规范 `feat/{module}-{description}`
- **先测试后合入**：PR 必须包含测试报告，QA 签署后方可合入 `main`
- **频繁 commit**：功能粒度的 commit，避免巨型 PR
- **每日集成**：DevOps 确保 `main` 分支随时可部署
